# Control Plane State Hygiene

TDK Control Plane separates code, contracts, and runtime memory so the repository stays stable while the operator workstation keeps its local truth.

## Code

Code is the executable system behavior:

- PowerShell scripts in `scripts/`
- Python runtime modules in `app/`
- tests and deterministic tooling

Code belongs in Git.

## Schema

Schema is the status contract that future UX and validators use as their source of structure.

Example:

```text
schemas/control_plane_status.schema.json
```

Schema belongs in Git because it defines the shape of valid runtime state.

## Runtime State

Runtime state is the current local snapshot generated by the operator workstation.

Example:

```text
state/control_plane_status.json
```

This file is local by default. It may contain workstation-specific status, timestamps, environment paths, Git sync status, and other operational facts that change from run to run.

## History Memory

History memory is the time-indexed archive of generated snapshots.

Example:

```text
state/history/control_plane_status_YYYYMMDD_HHMMSS.json
```

History memory is local by default. It represents the operator machine's timeline, not source code.

## Gitignore Rules

The repository ignores only runtime artifacts:

```gitignore
state/control_plane_status.json
state/history/*.json
```

The full `state/` directory is not ignored. This leaves room for future committed examples, fixtures, or documentation files.

## Fixture Snapshots

A snapshot can be committed deliberately when it is an example or test fixture, not live runtime state.

Use a separate fixture path, for example:

```text
tests/fixtures/control_plane_status.example.json
docs/examples/control_plane_status.example.json
```

Commit fixture snapshots only when they are stable, sanitized, and useful for tests, documentation, or UX development.
